- `apps/api/src/modules/prompt_template/input-dto/update-version.dto.ts`

### 엔드포인트

| Method | Path | 설명 |
|--------|------|------|
| GET | `/api/prompt-templates/list` | 전체 템플릿 목록 (버전 포함) |
| GET | `/api/prompt-templates/by-type/:type` | 타입별 조회 (enum 드롭다운) |
| GET | `/api/prompt-templates/:code/versions` | 코드 기준 버전 목록 |
| POST | `/api/prompt-templates/create` | 템플릿 + 초기 버전 A 생성 (트랜잭션) |
| PATCH | `/api/prompt-templates/:id/patch` | 템플릿 메타 수정 (name, description, tone_text) |
| DELETE | `/api/prompt-templates/:id/delete` | 템플릿 소프트 삭제 |
| POST | `/api/prompt-templates/:id/add-new-versions` | 버전 추가 (B, C, D 자동 태그) |
| PATCH | `/api/prompt-templates/:id/versions/:versionId/patch` | 버전 수정 |
| PATCH | `/api/prompt-templates/:id/versions/:versionId/activate` | 활성 버전 지정 |
| DELETE | `/api/prompt-templates/:id/versions/:versionId/delete` | 버전 소프트 삭제 |

### 주요 사항
- 생성 시 템플릿 + 버전 1개(A)를 트랜잭션으로 함께 생성
- 버전 태그는 version_no 기반 자동 부여 (1→A, 2→B, ... 27→AA)
- 활성 버전 지정 시 같은 템플릿 내 다른 버전은 자동 비활성
- 중복 type+code 생성 시 409 Conflict 반환
- Swagger에서 type은 enum 드롭다운 (`ARTICLE_AI`, `NEWSLETTER`)
- Zod 기반 입력 검증, Swagger example에 실제 목데이터 반영

## 2. 프롬프트 템플릿 편집 UI (프론트엔드)

### 구현 파일
- `apps/web/app/template/edit/page.tsx` — API 연결된 편집 페이지
- `apps/web/lib/api.ts` — 템플릿 API 함수 추가

### 기능
- 좌측 사이드바: 템플릿 목록 (타입 필터 + 검색)
- 우측 상단: 템플릿 이름, 설명, 톤 편집 (인라인)
- 버전 목록 탭: 카드 그리드 (2~3열), 활성 버전 표시, 삭제 확인
- 버전 편집 탭: prompt_text, few_shot_examples, parameters_json, change_note
- 미리보기 탭: 활성 버전의 프롬프트 읽기 전용
- 저장 (비활성) / 저장 + 활성 지정 분리
- 새 버전 만들기: 활성 버전 복사 후 생성
- 버전 태그 클릭으로 활성 버전 전환
- 타입 필터 변경 시 선택 자동 전환, 빈 상태 처리

### 관리자 Manage 버튼
- `apps/web/app/page.tsx`에 JWT role 디코딩 추가
- role === 'ADMIN'일 때 상단 오른쪽에 cherry 색 "Manage" 버튼 표시
- 클릭 시 `/template/edit`로 이동
- 데스크탑/모바일 헤더 모두 적용

## 4. 에이전트 통신 API (백엔드)

### 구현 파일
- `apps/api/src/modules/pipeline/agent-evaluation.service.ts` — 실행 패키지 조립 + 결과 저장
- `apps/api/src/modules/pipeline/input-dto/finish-evaluation.dto.ts` — 결과 저장 DTO
- `apps/api/src/modules/pipeline/input-dto/insert-article.dto.ts` — 기사 삽입 DTO
- `apps/api/src/modules/pipeline/pipeline.controller.ts` — 엔드포인트 추가
- `apps/api/src/modules/pipeline/pipeline.module.ts` — AgentEvaluationService 등록

### 에이전트용 엔드포인트

| Method | Path | 설명 |
|--------|------|------|
| POST | `/api/pipeline/agent/insert-article` | 기사 삽입 (URL 중복 스킵, 소스 자동 생성) |
| GET | `/api/pipeline/agent/ask-evaluation` | 평가 실행 패키지 (프롬프트 + 카탈로그 + 대기 기사) |
| POST | `/api/pipeline/agent/finish-evaluation` | 평가 결과 저장 (agent_json_raw) |

### ask-evaluation 응답 구조
```json
{
  "prompts": {
    "template_id": "uuid",
    "template_name": "기본 아티클 분석",
    "tone_text": "기술적이고 간결하게...",
    "versions": [
      { "version_id": "uuid", "version_tag": "A", "prompt_text": "...", ... },
      { "version_id": "uuid", "version_tag": "B", "prompt_text": "...", ... }
    ]
  },
  "catalog": {
    "pages": [
      {
        "page": "MODEL_UPDATES",
        "categories": [
          { "name": "OpenAI Family", "entities": ["GPT-4o", "GPT-5"] }
        ]
      }
    ],
    "side_categories": [{ "code": "CASE_STUDY", "name": "Case Study" }]
  },
  "items": [
    {
      "id": "ai-state-uuid",
      "article": { "title": "...", "content": "...", "url": "...", "published_at": "...", "source_name": "...", "source_type": "..." }
    }
  ]
}
```

### 설계 결정
- **카탈로그에서 UUID 제거** — name만으로 토큰 절약, 백엔드가 name→ID 역조회
- **version_tags 복수 지원** — `?version_tags=A,B`로 A/B 테스트 가능
- **훅 불필요** — 에이전트 30분 크론 pull + 백엔드 30분 크론 parse (15분 시차)

## 5. agent-json-parser 호환성 수정

### 변경 파일
- `apps/api/src/modules/pipeline/agent-json-parser.service.ts`

### 변경 내용
- `CROSS JOIN LATERAL` 패턴으로 entity 매칭 분리
- ID 있으면 (기존 포맷): PK 인덱스로 O(1) 매칭
- ID 없으면 (새 포맷): `lower(name)` + `entity_page`로 유니크 인덱스 역조회
- 기존 데이터 하위 호환 유지

## 전체 파이프라인 흐름

```
에이전트 크론 (매 :00, :30)
  1. POST /agent/insert-article     → 수집한 기사 저장
  2. POST /ingest-bulk               → user_article_state 생성
  3. POST /pregen-ai-state           → PENDING ai_state 생성
  4. GET  /agent/ask-evaluation      → 프롬프트 + 카탈로그 + 기사 수신
  5. LLM 호출                         → 평가 수행
  6. POST /agent/finish-evaluation   → 결과 저장

백엔드 크론 (매 :15, :45)
  7. POST /parse-agent-json          → agent_json_raw 파싱 → SUCCESS/FAILED
```
